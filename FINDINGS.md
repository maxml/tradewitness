# FINDINGS — TradeWitness

| # | Risk | Where | What | How to fix | Status |

| 1 | 🔴 | `apps/app/src/server/actions/trades.ts::updateTradeRecord` | IDOR (BOLA): updated trades by `tradeId` only, so any authenticated user could overwrite another user's trade by guessing the id. | Combine ownership predicate: `where(and(eq(TradeTable.id, tradeId), eq(TradeTable.userId, userId)))`. | ✅ fixed in [`82c23438`](../../commit/82c23438) |
| 2 | 🔴 | `apps/app/src/server/actions/trades.ts::deleteTradeRecord` | IDOR (BOLA): deleted trades by `recordId` only, no `userId` scope. | Resolve `auth()` and add `eq(TradeTable.userId, userId)` to the `where`. | ✅ fixed in [`11350139`](../../commit/11350139) |
| 3 | 🟡 | `apps/app/src/server/actions/trades.ts::createNewTradeRecord` | No `try/catch` around `db.insert`. A unique-violation or transient Postgres error surfaces as a 500 with no `{ error: true }` for the form. | Wrap insert in `try/catch`, return the existing error shape. | ✅ fixed in [`1760e241`](../../commit/1760e241) |
| 4 | 🟡 | `apps/app/package.json` | Outdated deps: `eslint@8.x` while Next.js 15 ships flat-config compatible `eslint-config-next` and the rest of the stack expects ESLint 9. | Bump `eslint` to `^9`, switch to flat config (`eslint.config.mjs`). | ✅ fixed in [`82c23438`](../../commit/82c23438) |
| 5 | 🟢 | `turbo.json` | Hardcoded values: `globalEnv` listed env names that no longer exist (`CLAUDE_API_KEY` vs the actual `ANTHROPIC_API_KEY`), so cache invalidation didn't track real env changes. | Re-derive `globalEnv` from `process.env` usages and `.env.example`. | ✅ fixed in [`82c23438`](../../commit/82c23438) |
| 6 | 🔴 | `apps/app/src/server/actions/archive.ts::deleteReportFromDB`, `::addRemoveFavorite` | IDOR: both functions looked up and mutated `ReportsTable` rows by `reportId` only, with no `userId` scope. Any authenticated user could toggle favorites or delete other users' AI reports. | Resolve `auth()`, short-circuit on missing `userId`, and pair every `eq(ReportsTable.id, …)` with `eq(ReportsTable.userId, userId)` inside `and(…)`. | ✅ fixed in [`b7718f99`](../../commit/b7718f99) |
| 7 | 🔴 | `apps/app/src/server/actions/strategies.ts::deleteStrategyFromDB`, `::editStrategy` | IDOR: `deleteStrategyFromDB` had no auth at all. `editStrategy` checked the caller's `userId` argument against `auth()` but then issued the `UPDATE` with `where(eq(id))` — so passing your own `userId` plus a victim's `strategyId` slipped past the equality check and overwrote their row. | Add `auth()` to delete; in both writes, scope the `where` with `and(eq(id), eq(userId))` so ownership is enforced at the SQL layer. | ✅ fixed in [`a65b5523`](../../commit/a65b5523) |
| 8 | 🟡 | `apps/app/src/server/actions/stripe.ts:19` | Non-null assertion `process.env.STRIPE_SECRET_API_KEY!` with no runtime guard. If the key is missing in a deployed env, the Stripe client constructs with `undefined` and the first call fails with an opaque error far from the root cause. | Add a runtime check at the top of the action and return `{ error: "Stripe not configured" }`, or fail fast on module load with a descriptive message. | 🔴 not yet (out of scope for this resubmission — flagged for follow-up) |

## Summary

- **5 of 8 findings are 🔴 critical IDOR/BOLA** — the codebase has a systemic pattern where Server Actions accept a row id and run the query with `where(eq(id, …))` without folding `userId` into the predicate. Fixed in `trades.ts`, `archive.ts`, `strategies.ts`. The same pattern check is now codified in `CLAUDE.md` Rule 4 to prevent regressions.
- **Finding #8 left open intentionally** — the fix touches deployment/observability concerns (env validation strategy across the monorepo) rather than a localized code change, and belongs in a follow-up PR.
